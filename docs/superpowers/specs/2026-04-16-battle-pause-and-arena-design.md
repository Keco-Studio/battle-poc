# Battle Pause And Arena Design

## 背景

本次调整围绕 `battle-poc` 的实时战斗面板展开，目标是同时解决四类问题：

- 战斗双方当前只能在左右半场内活动，缺少更自然的追逐与交错移动。
- 飘字停留偏久，战斗反馈节奏偏慢。
- 战斗生命值偏低，持续时间较短。
- 现有战斗逻辑依赖多个浏览器定时器，无法可靠支持“暂停后完全冻结、继续后原样恢复”。

## 目标

- 玩家与敌人在整个战斗画布内巡逻与追逐，不再受左右半场分界限制。
- 飘字淡出时间缩短到 0.5 秒。
- 开战时玩家与敌人的最大生命值、当前生命值都扩展为原来的 3 倍，保持当前比例一致。
- 在战斗标题区加入“暂停/继续”按钮。
- 暂停期间冻结战斗时间轴，避免攻击、演出、飘字、自动撤离等状态在后台继续消耗。
- 暂停期间显示半透明遮罩，明确反馈当前不可推进。

## 非目标

- 不改动战斗数值公式本身（攻击、防御、技能倍率、怪物相对强度倍率等），本次仅改动玩家基础生命常数 `BASE_STATS.hp` 以实现生命值 3 倍化。
- 不重构 Phaser 场景为外部状态机，仅做与本次需求直接相关的边界和表现调整。
- 不引入全新的全局时间系统到其他页面或非战斗模块。

## 方案概览

本次采用单一战斗时间轴方案：在 `BattlePanel.tsx` 中维护“战斗虚拟时钟”，所有战斗逻辑都以该时钟作为唯一的推进依据，而不是直接依赖 `Date.now()`、`setTimeout()` 或多个彼此独立的 `setInterval()`。

暂停时，虚拟时钟停止增长；继续时，从冻结时刻继续累计。这样可以天然保证：

- 玩家与敌人的攻击间隔不会在恢复瞬间补发。
- 战斗时长计数 `battleTimeSec` 会真正停止。
- `floatTexts`、`battleFx`、`heavyStrikePlaying`、中央战斗图标等表现状态不会在暂停期间自行结束。
- 自动撤离演出在暂停时同样冻结。

## 详细设计

### 1. 战斗虚拟时钟

在 `BattlePanel.tsx` 中新增一组围绕虚拟时间的状态与引用：

- `isPaused`：当前是否暂停。
- `battleClockMsRef`：当前累计的战斗虚拟毫秒数。
- `lastRealTickAtRef`：最近一次主循环执行时的真实时间戳。

主循环仍可以保留单个较高频率的浏览器轮询，但循环内部不再把真实时间直接作为业务时间，而是：

1. 读取当前真实时间。
2. 计算距离上次循环经过了多少真实毫秒。
3. 若未暂停，则把这段增量累加到 `battleClockMsRef`。
4. 所有战斗判定统一读取 `battleClockMsRef.current`。

这样，暂停时只需阻止虚拟时间累加，而不必分别停止每个具体行为的浏览器定时器。

### 2. 到期事件统一改为基于截止时刻

所有“持续一段时间后结束”的战斗表现，统一改为“记录在虚拟时间上的结束时刻”。

具体包括：

- `battleFx`：增加 `battleFxUntilRef`，当当前虚拟时间到达截止时刻后恢复为 `none`。
- `heavyStrikePlaying`：增加 `heavyStrikeUntilRef`，到点后关闭。
- `showCenterBattleIcon`：增加 `centerIconUntilRef`，到点后隐藏。
- `floatTexts`：每条记录增加 `expireAt`，渲染层只消费未过期数据。
- `autoFleeAnimating`：增加 `autoFleeUntilRef`，到点后触发逃跑收尾。

这样，暂停时无需暂停浏览器 `setTimeout()`，因为这些状态是否结束完全由虚拟时间决定。

### 3. 攻击与战斗计时推进

玩家与敌人的攻击触发时刻改为直接存储在虚拟时间轴上：

- `nextPlayerAtkAtRef`
- `nextEnemyAtkAtRef`

当虚拟时间达到对应时刻时才执行一次攻击结算，并重新写入下一次触发的虚拟时刻。暂停期间由于虚拟时间不前进，因此攻击不会继续触发。

`battleTimeSec` 则改为从虚拟时间派生：

- 使用 `Math.floor(battleClockMsRef.current / 1000)` 同步到界面。
- 暂停时不增长，继续后无缝接续。

### 4. 暂停交互

在 `BattlePanel.tsx` 顶部标题区域增加按钮：

- 未暂停时显示“暂停”
- 暂停时显示“继续”

交互规则：

- 仅在战斗进行中显示；若战斗已结束，则不再显示。
- 自动撤离演出期间仍允许显示当前状态，但若实现上与流程冲突，则以“不允许在自动撤离期间手动暂停”为准，避免交叉状态复杂化。
- 点击后只切换 `isPaused`，不直接重置任何战斗状态。

视觉反馈：

- 在战斗画布上叠加 `rgba(0, 0, 0, 0.4)` 遮罩。
- 可在中央显示“已暂停”提示，作为辅助反馈，但不是硬性需求。

### 5. 战斗生命值 3 倍化

通过修改底层战斗属性常量 `BASE_STATS.hp`（`app/constants.ts` 第 94 行）从 `30` 改为 `90`，使玩家基础生命值直接变为 3 倍。怪物最大生命由 `calcEnemyStats()` 基于同等级玩家 hp × `MONSTER_VS_PLAYER_STAT_MULT`（1.2）计算得出，因此玩家基础 hp 改动后怪物最大生命同步等比放大，无需单独修改。

改动前 Lv1 角色基础: `30` → 改动后: `90`（90 / 30 = 3x）
改动前 Lv1 怪物最大: `(30 + 0×10) × 1.2 = 36` → 改动后: `(90 + 0×10) × 1.2 = 108`（108 / 36 = 3x）

此修改影响所有基于 `calcPlayerStats()` / `calcEnemyStats()` 计算最大生命的场景，包括战斗外角色面板、背包、存档等。若某处需要显示”原始”（未 3 倍化）的生命值，需要显式使用未乘以倍率的版本，不在此 spec 范围内处理。

非目标：不改动生命值相关的派生计算（如经验值升级曲线、装备加成、回复量等），仅改动基础生命常数。

### 6. Phaser 战斗场景移动边界

`BattleScene.ts` 中移除当前左右半场 `ZONE` 设计，改为单一全局边界：

- `xMin: 90`
- `xMax: W - 90`
- `yMin: 120`
- `yMax: H - 120`

相应调整：

- 初始站位仍保持左右相对分布，避免开场重叠。
- `pickPatrolTarget()` 直接在全局边界内采样。
- `moveToward()` 的 `bounds` 参数使用统一全局矩形。
- `nudgeChaseTowardOpponent()` 中的目标拉拽也基于全局边界。
- 冲锋、击退、逃跑等需要钳制位置的逻辑统一改用全局边界，避免残留左右半场常量。

### 7. 飘字淡出时长

`BattleScene.ts` 的 `spawnFloatText()` 中，将 tween 的 `duration` 从 `820` 改为 `500`。

同时，`BattlePanel.tsx` 中飘字数据的虚拟生命周期应与渲染侧表现保持一致，避免 React 状态比 Phaser tween 更晚清除太多。允许略留缓冲，但不应明显长于 0.5 秒。

## 数据流

1. `BattlePanel.tsx` 维护战斗业务状态、虚拟时钟与 UI 状态。
2. `battleVisualRef.current` 从业务状态派生出当前需要渲染给 Phaser 的快照。
3. `BattlePhaserCanvas` / `BattleScene.ts` 只消费快照，不负责暂停业务逻辑。
4. `BattleScene.ts` 内仍可保留自身 tween，但上层不会在暂停期间继续推进会过期的业务状态。

## 错误处理与边界情况

- 若在暂停前一帧正好触发攻击，则该攻击按已进入结算处理，不回滚。
- 若战斗结束，则自动解除暂停态或忽略暂停按钮，以避免结束遮罩与暂停遮罩叠加冲突。
- 若战斗关闭后重新打开，虚拟时钟和所有截止时刻必须重置，避免继承上一场战斗的到期事件。
- 飘字按 `id` 与 `expireAt` 双重管理，避免恢复后旧飘字重复处理。

## 测试策略

优先增加聚焦单测，验证单一战斗时间轴的核心行为：

- 暂停期间虚拟时钟不推进。
- 恢复后虚拟时钟继续推进。
- 基于虚拟时间的到期事件不会在暂停时结束。
- 基于虚拟时间的攻击调度不会在恢复时补发多次。

此外进行项目级验证：

- 运行 `npx vitest run`

## 实施顺序

1. 为 `BattlePanel.tsx` 提炼或引入虚拟时钟与截止时刻模型。
2. 先用测试覆盖暂停/继续的时间推进行为。
3. 把攻击、飘字、表现态、开场图标、自动撤离切到虚拟时间轴。
4. 加入暂停按钮与遮罩。
5. 修改 `app/constants.ts` 中 `BASE_STATS.hp` 为 90，实现全局生命值 3 倍化。
6. 修改 `BattleScene.ts` 的全局移动边界与飘字时长。
7. 跑完整 Vitest 验证。

## 风险与缓解

- 风险：`BattlePanel.tsx` 当前逻辑集中，改动时容易遗漏某个仍依赖真实时间的状态。
  缓解：把所有“持续时长”集中整理，逐项替换成虚拟截止时刻。

- 风险：改动 `BASE_STATS.hp` 会同时影响战斗外所有依赖 `calcPlayerStats()` 的场景（如角色面板、背包显示等）。
  缓解：本次接受这一影响；如有需要，后续可在需要"原始基础值"的场景使用未乘以常数的独立常量。

- 风险：Phaser 内部 tween 仍按真实时间运行。
  缓解：本次仅要求暂停不消耗上层业务状态；若发现 Phaser 内部演出在暂停时仍明显推进，再补充对场景 tween/time 的显式暂停控制，但这不是首选方案。
